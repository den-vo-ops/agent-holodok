### Фаза 1B — Обработка ошибок LLM (НОВОЕ, сквозная)

**Цель:** централизованно перехватывать любые ошибки Claude API, показывать мастеру понятный русский текст и писать технические детали в лог. Ни один путь не должен приводить к молчанию бота.

**Архитектурное решение:** централизованно в клиенте. `ClaudeClient` ловит исключения Anthropic SDK → логирует трейсбек → бросает доменное `LLMError` с готовым русским `.user_message`. Хендлеры ловят `LLMError` и отправляют `.user_message`. Глобальный error-handler aiogram — сеть безопасности для всего непойманного.

| № | Задача | Файлы | Тесты | Зависит от | Статус |
|---|---|---|---|---|:---:|
| 1B-1 | Доменная ошибка + маппинг | `holodok_agent/llm/errors.py` | `tests/test_errors.py` | 1A-1 | ✅ |
| 1B-2 | Обёртка вызова в клиенте (лог + `raise LLMError`) | `holodok_agent/llm/client.py` | `tests/test_client.py` (доп.) | 1B-1, 1A-5 | ✅ |
| 1B-3 | Перехват `LLMError` в хендлерах → русский текст | `holodok_agent/bot/handlers.py` | `tests/test_handlers.py` (доп.) | 1B-2, 1A-10 | ✅ |
| 1B-4 | Глобальный error-handler aiogram (сеть безопасности) | `holodok_agent/bot/main.py` | ручной smoke (неверный ключ) | 1B-3, 1A-11 | ✅ |

**Детали задач:**

- **1B-1** `holodok_agent/llm/errors.py`:
  - `class LLMError(Exception)` с полем `.user_message: str` (готовый русский текст для пользователя). Технический текст — в аргументе исключения.
  - `to_llm_error(exc: Exception) -> LLMError` — маппит исключения Anthropic SDK (`RateLimitError`, `APITimeoutError`, `APIConnectionError`, `AuthenticationError`, `APIStatusError`, прочее) на русские сообщения из таблицы §3.
  - Тест: каждому типу исключения — ожидаемый русский текст; неизвестному исключению — общий fallback.

- **1B-2** доработка `holodok_agent/llm/client.py`:
  - `complete()` оборачивает `self._client.messages.create(...)` в `try/except anthropic.APIError as exc`.
  - В `except`: `logger.exception("Claude API call failed")` (полный трейсбек и детали разработчику), затем `raise to_llm_error(exc) from exc`.
  - Тест: мок SDK бросает `RateLimitError` → `complete()` бросает `LLMError` с текстом про «слишком много запросов»; факт логирования проверяется через `caplog`.

- **1B-3** доработка `holodok_agent/bot/handlers.py`:
  - Обернуть `analyze_style(...)` в `handle_onboarding_done` и `generate_content(...)` в `_generate_and_send` в `try/except LLMError as exc:` → `await message.answer(exc.user_message)` и корректно завершить (сбросить/сохранить состояние FSM так, чтобы мастер мог повторить).
  - Тест: мок бросает `LLMError` → пользователю уходит `.user_message`, а не трейсбек; черновик не помечается использованным при ошибке.

- **1B-4** глобальный error-handler в `holodok_agent/bot/main.py`:
  - Зарегистрировать `dp.errors.register(...)`: ловит ЛЮБОЕ непойманное исключение при обработке апдейта → `logger.exception(...)` + попытка отправить владельцу общий русский текст (fallback из §3).
  - Ручной smoke: временно подставить неверный `ANTHROPIC_API_KEY`, вызвать «Дай идею» → бот отвечает по-русски, а не молчит; в логах — трейсбек.

**Критерии готовности 1B:**
- [x] При любой ошибке LLM мастер получает понятный русский текст (не технический) — проверено поведенчески (маппинг + скрытие техтекста).
- [x] В логах — полный трейсбек с типом исключения и сценарием (`logger.exception` в клиенте и глобальном хендлере).
- [x] Нет пути «ошибка → молчание бота»: клиент → `LLMError` → хендлер, плюс глобальный error-handler aiogram как сеть безопасности. Живой smoke с неверным ключом — **ожидает владельца**.
- [x] Доп. тесты `test_errors.py` / расширения `test_client.py`, `test_handlers.py` — зелёные (итого 60 passed).

## 3. Русские сообщения об ошибках LLM (источник правды для текстов)

Тексты, которые видит мастер, и что при этом пишется в лог. Используются в `to_llm_error()` (1B-1). Тон — спокойный, без техножаргона, всегда с подсказкой «что делать».

| Тип ошибки (Anthropic SDK) | Причина | Текст пользователю (RU) | В лог |
|---|---|---|---|
| `RateLimitError` | Превышен лимит запросов | «Слишком много запросов к ИИ подряд. Подождите минуту и попробуйте снова.» | тип, `repr(exc)`, трейсбек, сценарий |
| `APITimeoutError` / `APIConnectionError` | Таймаут / нет связи с API | «Не удалось связаться с ИИ — похоже, проблемы с сетью. Попробуйте ещё раз через минуту.» | то же |
| `AuthenticationError` | Неверный/просроченный ключ | «Не получается обратиться к ИИ из-за настроек доступа. Я записал проблему в журнал — попробуйте позже.» | то же + пометка «проверить ANTHROPIC_API_KEY» |
| `APIStatusError` (5xx) | Сбой на стороне сервиса | «ИИ временно недоступен. Попробуйте ещё раз чуть позже.» | то же + код статуса |
| `BadRequestError` / 4xx (кроме auth) | Некорректный запрос/слишком длинный ввод | «Не получилось обработать запрос — возможно, текст слишком длинный. Попробуйте короче.» | то же |
| Прочее / неизвестное | Любая иная ошибка | «Извините, не получилось подготовить текст. Попробуйте ещё раз через пару минут.» | то же (общий fallback) |

**Правило логирования (везде):** `logger.exception(...)` с типом исключения, `repr(exc)`, полным трейсбеком и названием сценария/действия. Пользователю технический текст **не показывается никогда**.

## 5. Зависимости между задачами

**Фаза 1A (внутренний порядок из плана ядра):**
```
1A-1 (каркас)
 ├─ 1A-2 config
 ├─ 1A-3 db ─────────────┐
 ├─ 1A-4 auth ───────────┤
 ├─ 1A-5 client ─┬─ 1A-6 style ─┤
 │               └─ 1A-8 content┤
 ├─ 1A-7 rules ──────────┤
 └─ 1A-9 keyboards ──────┤
                         └─ 1A-10 handlers ─ 1A-11 scheduler ─ 1A-12 deploy
```

**Фаза 1B вплетается в 1A:**
- `1B-1` ← 1A-1
- `1B-2` ← 1B-1 + 1A-5 (дорабатывает client сразу после его создания)
- `1B-3` ← 1B-2 + 1A-10 (дорабатывает handlers)
- `1B-4` ← 1B-3 + 1A-11 (глобальный handler в main.py)

**Между фазами:**
- `1C` ← 1A + 1B (нужны db, клиент с обработкой ошибок, бот и планировщик)
- `1D` ← Фаза 0 (go) + 1A + 1B
- `2` ← 1C
- `3` ← gate-проверка

**Критический путь MVP:** 1A-1 → 1A-3/5 → 1A-6/8 → 1A-10 → 1B-2/3 → (1A-11, 1B-4) → 1A-12. Модули 1C/1D наращиваются поверх готового ядра.

## 6. Как обновлять этот файл (регламент автообновления)

Этот план — рабочий трекер, а не архив. Он обязан отражать реальность.

**После завершения каждой задачи:**
- Обновить её строку в **таблице фазы** (§2) и в **общей таблице прогресса** (§4): значок статуса + столбец «Факт» (что реально сделано, кратко).

**После завершения фазы:**
- Отметить чек-боксы «Критерии готовности» фазы.
- Пересмотреть блокеры зависимых фаз: если предпосылка снята — заменить ⛔ на ⬜ (например, Фаза 0 «go» разблокирует 1D).
- Обновить статус фазы в §1 и дату «Последнее обновление» в шапке.

**Связь с CLAUDE.md §9:**
Любая сессия, завершившая задачу или фазу, **обязана обновить Plan.md перед завершением работы** — не оставлять «на потом». Это то же правило самообновления, что действует для CLAUDE.md.

**При создании детального плана подсистемы** (1C, 1D, 2, 3) через скилл `writing-plans`:
- Разбить крупную задачу здесь на подзадачи со ссылкой на новый детальный план (как сделано для 1A → план ядра).

**Разовый шаг (вне plan-mode):** добавить в CLAUDE.md §2 и §7 упоминание `Plan.md` как рабочего трекера прогресса, дополняющего детальные планы в `docs/superpowers/plans/`.
