# CLAUDE.md — Holodok Agent

## 1. О проекте

Персональный ИИ-агент в Telegram для одного мастера по ремонту холодильников (Симферополь), который сейчас существует только на сарафанном радио и без онлайн-присутствия. Агент следит за конкурентами и горячими лидами, готовит контент (посты/объявления/ответы на отзывы) в личном стиле мастера и присылает еженедельную аналитику. Разработка ведётся Claude (Opus) напрямую как обычный код — без no-code платформ (n8n/Make).

**Статус:** реализованы Фазы 1A (ядро агента) и 1B (обработка ошибок LLM) — 61 автотест зелёный, влито в `master`. **LLM-провайдер: Groq** (по решению владельца от 2026-07-04 перешли с Claude/Anthropic на Groq; переменные окружения — `BOT_TOKEN`, `ADMIN_IDS`, `GROQ_API_KEY`, `GROQ_MODEL`, `DB_PATH`). Осталось: деплой на VPS, затем Фазы 1C/1D/2/3 (см. Plan.md). Подробности статусов — в Plan.md §4.

> ⚠️ Расхождение со spec.md: спецификация как «источник правды» всё ещё требует Claude/Opus (§8). Смена на Groq сделана прямым решением владельца в чате; spec.md следует привести в соответствие через скилл `discovery-interview` при следующем раунде требований.

## 3. Tech Stack

| Слой | Технология | Почему |
|---|---|---|
| Язык | Python 3.11+ | стандарт для скриптов/ботов, легко деплоить на VPS |
| Telegram-фреймворк | aiogram >=3.4,<4 | async, встроенный FSM для многошаговых диалогов (онбординг, сценарии) |
| LLM | **Groq API** (SDK `groq`), модель `llama-3.3-70b-versatile` по умолчанию (конфигурируемо через `GROQ_MODEL`) | решение владельца от 2026-07-04 — перешли с Claude на Groq (см. §1). Изначально spec.md требовал Claude/Opus; расхождение зафиксировано, spec.md пока не переписан |
| Хранилище | SQLite (stdlib `sqlite3`) | один пользователь, низкая нагрузка — отдельная СУБД не нужна |
| Планировщик | APScheduler >=3.10,<4 | ежедневный сбор данных + еженедельный отчёт + ежемесячный опрос метрик |
| Тесты | pytest + pytest-asyncio | TDD, `asyncio_mode = auto` в `pytest.ini` |
| Хостинг | существующий VPS владельца, systemd | бюджет ограничен (~2000–3000₽/мес суммарно с LLM) |

## 4. Архитектура

```
Мастер (Telegram)
      │
      ▼
aiogram Router/Handlers (bot/handlers.py)   ← FSM-состояния: Onboarding, ScenarioFlow, MetricsFlow
      │                          │
      ▼                          ▼
llm/style.py, llm/content.py     db.py (SQLite)
(бизнес-логика, чистые функции)  (стиль, правила, черновики, метрики)
      │
      ▼
llm/client.py → Anthropic API (Claude Opus)
```

Бизнес-логика (анализ стиля, сборка промптов, парсинг команд) намеренно вынесена из aiogram-хендлеров в чистые функции без побочных эффектов — именно они покрываются юнит-тестами. Хендлеры — тонкий связующий слой.

## 5. Пять ключевых решений

1. **Кастомный Python-код вместо n8n/Make.** Разработку ведёт Claude (Opus) напрямую — no-code платформа не нужна, когда есть «программист». Даёт полный контроль над бизнес-логикой стиля/памяти.
2. **SQLite, не PostgreSQL/etc.** Один пользователь, скромный объём данных — лишняя инфраструктура не оправдана (YAGNI).
3. **Уведомлять о ЛЮБОМ изменении цены конкурента, без порога.** Осознанный выбор владельца, хотя research.md/critique.md рекомендовали порог. Риск шума задокументирован в spec.md §10, порог сделан technically-configurable на будущее.
4. **«Ответ на отзыв» — только по вручную вставленному тексту.** Автоматический мониторинг собственной репутации мастера в скоуп не входит — упрощает MVP, но ограничивает функцию.
5. **Обучение стилю — через загрузку старых текстов, без анкеты.** Агент анализирует тон/лексику/структуру реальных объявлений мастера при первом /start, а не собирает ответы на вопросы.

## 6. Тестирование

- Тесты лежат в `tests/`, зеркалируя модули `holodok_agent/` один-в-один (`db.py` → `tests/test_db.py`).
- Подход: TDD — сначала падающий тест, потом минимальная реализация (см. план в `docs/superpowers/plans/`).
- Async-тесты не требуют декоратора `@pytest.mark.asyncio` (`asyncio_mode = auto` в `pytest.ini`).
- Хендлеры aiogram тестируются вызовом функций напрямую с `MagicMock`/`AsyncMock` вместо поднятия настоящего Dispatcher.

**Чек-лист перед коммитом:**
- [ ] `python -m pytest` — всё зелёное
- [ ] Новый код не содержит TODO/заглушек
- [ ] Для aiogram-хендлеров, которые физически нельзя покрыть unit-тестом (сетевые вызовы Telegram/Claude) — пройден ручной smoke-тест из плана (Task 10/12)

## 8. Commands

```bash
# Установка
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # заполнить TELEGRAM_BOT_TOKEN, OWNER_TELEGRAM_ID, ANTHROPIC_API_KEY

# Тесты
python -m pytest -v

# Локальный запуск бота
set -a; source .env; set +a
python -m holodok_agent.bot.main

# Деплой на VPS (systemd)
sudo cp -r . /opt/holodok-agent && cd /opt/holodok-agent
python3 -m venv .venv && .venv/bin/pip install -r requirements.txt
sudo cp deploy/holodok-agent.service /etc/systemd/system/
sudo systemctl daemon-reload && sudo systemctl enable --now holodok-agent
journalctl -u holodok-agent -f   # логи
```

## 9. Самообновление

Этот файл — не архив, а рабочая карта проекта. Он должен переписываться, а не молча устаревать.

| Событие | Что обновить в CLAUDE.md |
|---|---|
| Завершён очередной план из `docs/superpowers/plans/` | §2 (структура — заменить «план» на факт), §3/§4 если появился новый компонент стека или слой архитектуры |
| Создан новый план (новая подсистема: Авито/Юла, отчёты, горячие лиды) | §2 (добавить будущие модули), таблица §7 (новая строка на план) |
| `spec.md` изменён через `discovery-interview` | §1, §5 (ключевые решения могли измениться) |
| Изменился стек/библиотека | §3 |
| Изменилось решение из спорных компромиссов (например, добавлен порог на уведомления) | §5 |

Правило: любая сессия, которая заканчивает задачу, меняющую структуру, стек или ключевое решение, обязана перед завершением обновить соответствующий раздел этого файла — не оставлять это «на потом».
